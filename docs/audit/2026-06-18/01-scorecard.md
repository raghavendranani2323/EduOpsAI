# EduOps AI Production Scorecard

Scores are 0-10. A score of 10 means production-ready for paying Indian institutions in that area. Scores reflect confirmed repository, runtime, browser, DB/RLS, and market evidence from 18/06/2026.

| Area | Score | Evidence | To Reach Next Level |
|---|---:|---|---|
| Product completeness | 5 | Core modules exist across students, classes, attendance, fees, admissions, homework, notices, exams, timetable, parent portal, settings, and exports. Several are incomplete or disconnected, especially online parent payments, communications, offline sync, audit logs, and academic year rollover. | Finish the selected ICP's complete vertical slices: attendance, fees, WhatsApp reminders, admissions conversion, homework/notices, parent view, imports, and reports. |
| Mobile UX | 6 | Dashboard and sampled routes rendered without horizontal overflow at 320-412 widths. Attendance marking is phone-oriented. Status picker buttons measured 40px on 360px width, below 44px target. Offline and large-list workflows are not proven. | Increase tap targets, test large classes/invoices, add bottom-sheet ergonomics, and verify one-handed fee/attendance flows on low-end Android. |
| Desktop UX | 6 | Desktop routes render cleanly and use a familiar SaaS shell. Some modules are empty/static and reporting depth is low. | Add dense operational views, better filters, reports, and support/admin diagnostics without bloating the mobile path. |
| Visual design | 6 | Authenticated app looks credible, restrained, and consistent in sampled routes. Landing page includes unproven claims and generic promotional patterns. Empty states are sometimes thin. | Tighten copy, trust signals, status design, empty states, and real product screenshots. Remove unsupported payment/offline claims. |
| Accessibility | 4 | Keyboard/a11y not fully automated. Issues found: no skip link, Telugu text with `lang="en"`, 40px attendance controls, possible unlabeled icon links, lint error on `children` prop. | Add axe/Playwright a11y tests, fix labels/focus/touch targets/language, verify dialogs and 200 percent zoom. |
| Performance | 6 | `pnpm build` passed. Sampled routes had no console errors. App has many client components and React Hook Form compiler warnings. No Lighthouse/CWV numbers captured. | Measure production CWV, reduce route JS, paginate large lists, inspect query plans, and add performance budgets. |
| Reliability | 4 | Build/type/RLS pass, but lint fails. Offline sync is partial. Payment and receipt flows have race/idempotency risks. No health check, monitoring, runbooks, or restore evidence. | Add idempotency, transactions, retries, observability, runbooks, and failure-mode tests. |
| Security | 4 | RLS exists and passed smoke tests. Critical concerns remain in attendance ID validation, push auth, CSV export, public uploads, plaintext password sharing, error leakage, security headers, and parent bearer links. | Close P0/P1 security findings and add automated authz/tenancy tests. |
| Tenant isolation | 6 | Main tables have RLS policies, `withRls` is common, and RLS test passed for seeded resources. Attendance records can reference unvalidated students. `prismaAdmin` usage requires continuous review. | Validate every ID against institution and resource scope, expand RLS tests to all routes/tables/tokens/exports/webhooks. |
| Role permissions | 5 | Role-aware navigation exists. Teacher class list is scoped. Server-side teacher scope is missing on attendance detail/API. Teachers can likely call APIs hidden from UI in some modules. | Build a formal permission matrix and enforce it in every API/action, not only in navigation. |
| Database design | 6 | Broad schema, many relations, money uses Decimal, RLS enabled. Missing or weak uniqueness/indexes for admissions, messages, homework/notices, invoice dedupe, student admission numbers, guardian duplication. | Add tenant-scoped constraints, query indexes, archival semantics, and migration tests. |
| API quality | 5 | Many APIs use Zod and `requireInstitution`. Gaps: rate limits, pagination, idempotency, role checks, output validation, safe error messages, and audit logging. | Add shared authz, safe error format, idempotency keys, transactions, and API tests. |
| Offline/PWA | 3 | Manifest and icons exist. `public/sw.js` handles push only; no fetch interception. IndexedDB queue exists but is limited, app-open sync only, no conflict handling, and stale cache bug returns expired data. | Truthfully label as "limited draft-safe attendance queue" or implement robust offline attendance sync with conflict UI and logout cleanup. |
| Payments | 3 | Fee plans, invoices, payments, receipt and Razorpay APIs exist. Parent online pay says coming soon. Webhook reconciliation and manual payment concurrency are insufficient. | Complete parent Razorpay flow, order storage, signature/order/amount reconciliation, idempotency, receipts, refunds/voids, reports, and tests. |
| Parent experience | 4 | Token portal and OTP parent login exist. Parent page is mobile-friendly in code structure. Payment not usable, token is long-lived, trust/privacy language limited, no robust multi-child/consent story. | Add secure revocable parent access, payment, read states, consent/privacy copy, and low-bandwidth UX tests. |
| Teacher experience | 6 | Teacher nav lockdown exists and attendance/homework/notices are practical. Server-side teacher scope gap is serious. | Fix teacher API scope, add assigned-class workflows, offline attendance reliability, and teacher acceptance tests. |
| Accountant experience | 4 | Fees module exists and accountant role is present. Financial reporting, reconciliation, receipt cancellation, ageing, collector tracking, and payment-mode summaries are incomplete. | Build daily collection, outstanding ageing, receipt audit, mode reports, and accountant E2E tests. |
| Owner/admin experience | 5 | Dashboard, settings, team, academic year, onboarding checklist, and modules exist. Setup dependencies and blank-state guidance need stronger activation path. | Add guided setup, sample data, import templates, support hooks, and clear success metrics. |
| Onboarding | 4 | Signup/onboarding routes exist. Browser session showed onboarding checklist. Import templates/guidance and first-value path are incomplete. | Reduce time to first attendance/fee reminder, add checklist telemetry, demo data, and migration support. |
| Landing-page conversion | 3 | Public landing exists, but existing auth redirected `/` to dashboard in local session. Copy claims UPI/Razorpay/offline features that are not complete. Missing pricing clarity, social proof, contact, privacy, FAQ depth, screenshots. | Narrow ICP, make truthful claims, add real screenshots, WhatsApp CTA, pricing, trust/legal/support pages. |
| SEO | 3 | Basic metadata and manifest exist. No `robots.txt`, `sitemap.xml`, canonical, OG/Twitter images, structured data, or local/segment pages found. | Add technical SEO basics and content for coaching/tuitions/schools without misleading feature claims. |
| Observability | 2 | No health check, request IDs, structured logs, Sentry/OpenTelemetry, uptime checks, webhook monitoring, or incident runbooks found. | Add production observability before paid pilots. |
| Testing | 3 | `typecheck`, `build`, and RLS tests exist. `lint` fails. No broad E2E, API, a11y, payment, offline, visual, or load suite found. | Fix lint and add high-value Playwright/API/RLS/payment/offline tests. |
| Compliance readiness | 3 | Audit logs table exists but sparse. No visible privacy/terms/retention/deletion/consent workflow. DPDP/CERT-In readiness needs legal review. | Add privacy notice, consent, retention, deletion/export, vendor inventory, breach/runbook, and legal review. |
| Customer support readiness | 2 | No help centre, support workflow, status page, support access controls, or operational diagnostics found. | Add WhatsApp/email support, help docs, issue reporting, error reference IDs, and tenant support tooling. |
| Monetisation readiness | 2 | No EduOps subscription billing, plan limits, trial, grace period, upgrade/downgrade, institution invoices, or feature gating found. | Define pricing, trial, billing, plan enforcement, cancellation/export, and founder-led onboarding motion. |
| Competitive differentiation | 5 | Strong potential: lightweight mobile-first Indian coaching/tuitions workflow. Current implementation cannot yet claim reliable payments/offline/communications. | Focus on speed, WhatsApp-first workflows, low setup effort, transparent pricing, and trustworthy data/security. |
